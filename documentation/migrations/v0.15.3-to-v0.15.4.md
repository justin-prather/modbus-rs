# Migration Guide: v0.15.3 to v0.15.4

This migration guide details the breaking changes, API improvements, and package restructuring introduced in `v0.15.4`.

---

## 1. Directory & Package Layout Restructure

To improve package maintainability and publish cleanliness, all build artifacts have been consolidated under a unified `dist/` output folder:

- **Moved Sub-packages**: The native platform binding stubs and the WebAssembly binding package have been moved from the root `npm/` directory into `dist/npm/`:
  - `npm/wasm/` $\rightarrow$ `dist/npm/wasm/`
  - `npm/win32-x64-msvc/` $\rightarrow$ `dist/npm/win32-x64-msvc/`
  - (and all other native platform folders).
- **Eliminated `src/` wrappers**: Hand-written wrapper entry files (`src/index.browser.js`, `src/index.web.js`, etc.) have been removed. These wrappers are now generated dynamically at build time inside the `dist/` directory.

### Migration Action
If you were resolving or referencing the sub-packages directly from local paths (e.g. in workspaces, monorepos, or package dependencies), you must update your reference paths:

```diff
  "devDependencies": {
-   "modbus-rs-wasm": "file:./npm/wasm"
+   "modbus-rs-wasm": "file:./dist/npm/wasm"
  }
```

---

## 2. API Alignment & Type Parity

To establish complete cross-platform type parity between the Node.js native client and the browser WebAssembly client, several method signatures have been updated.

### A. Strict Typed Array APIs
All register read/write operations now utilize `Uint16Array` instead of raw Javascript `number[]` arrays to align with standard buffer structures and improve performance.

#### Method Return Types:
- `readHoldingRegisters()` returns `Promise<Uint16Array>` instead of `Promise<number[]>`.
- `readInputRegisters()` returns `Promise<Uint16Array>` instead of `Promise<number[]>`.
- `readWriteMultipleRegisters()` returns `Promise<Uint16Array>` instead of `Promise<number[]>`.

#### Method Parameter Types (WASM Client):
- `writeMultipleRegisters(options)` $\rightarrow$ `options.values` must be a `Uint16Array` (raw array support removed).
- `readWriteMultipleRegisters(options)` $\rightarrow$ `options.writeValues` must be a `Uint16Array` (raw array support removed).
- `writeFileRecord(options)` $\rightarrow$ `options.requests[].recordData` must be a `Uint16Array` (raw array support removed).
- `diagnostics(options)` $\rightarrow$ `options.data` must be a `Uint16Array` (raw array support removed).

#### Code Snippet Example:
```javascript
// Before
await client.writeMultipleRegisters({ address: 100, values: [5, 6, 7] });

// After
await client.writeMultipleRegisters({ address: 100, values: new Uint16Array([5, 6, 7]) });
```

---

### B. `readFifoQueue` Signature Alignment
The WebAssembly `readFifoQueue` method return signature has been aligned with the Node.js native binding definition:

- **WASM `readFifoQueue()`** return type changed from `Promise<Uint16Array>` to `Promise<FifoQueueResponse>`.

```typescript
export interface FifoQueueResponse {
  /** Number of elements in the FIFO queue. */
  count: number;
  /** Array of 16-bit FIFO register values. */
  values: Uint16Array;
}
```

#### Code Snippet Example:
```javascript
// Before (WASM)
const registers = await client.readFifoQueue({ address: 42 });
const len = registers.length;

// After (WASM & Node)
const response = await client.readFifoQueue({ address: 42 });
const len = response.count;
const registers = response.values;
```

---

### C. Added `isConnected()` state verification
To allow checking of active transport client status without attempting a request, the `isConnected()` method has been exposed across both native and WASM serial/TCP clients:

- `WasmSerialModbusClient.isConnected(): boolean` (Newly added)
- `WasmWsModbusClient.isConnected(): boolean`
- `AsyncTcpModbusClient.isConnected(): boolean`
- `AsyncSerialModbusClient.isConnected(): boolean`

---

### D. Typed Server Handlers (NodeJS)
NodeJS server binding methods now natively generate strongly typed `ServerHandlers` instead of generic `object` arguments:

```diff
- bind(options: TcpServerOptions, handlers: object): Promise<AsyncTcpModbusServer>;
+ bind(options: TcpServerOptions, handlers: ServerHandlers): Promise<AsyncTcpModbusServer>;
```

#### Server Handler Return Type Updates:
Server handlers returning registers must return a `Uint16Array` instead of a raw JavaScript number array:
- `onReadHoldingRegisters` return value updated to `Uint16Array`.
- `onReadInputRegisters` return value updated to `Uint16Array`.

---

### E. Property Syntax Parity (`pendingRequests`)
To match the WASM read-only property naming conventions, the native Node.js class properties have been aligned:

- Changed from getter `get pendingRequests(): boolean` to property declaration `readonly pendingRequests: boolean`.
