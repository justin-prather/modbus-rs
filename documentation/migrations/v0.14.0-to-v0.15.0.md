# Migration Guide: modbus-rs Node.js FFI (v0.14.0 → v0.15.0)

This guide documents the breaking changes and new APIs introduced in **v0.15.0** of the `modbus-rs` Node.js bindings, and how to update your codebase.

---

## 1. TCP Connect Option Rename: `timeoutMs` → `requestTimeoutMs`

To unify client transport options with the RTU and ASCII serial transports, the request timeout option has been renamed.

### Action Required
Update any calls to `AsyncTcpTransport.connect(...)` replacing the `timeoutMs` option with `requestTimeoutMs`.

```js
// ❌ v0.14.0 (Old)
const transport = await AsyncTcpTransport.connect({
  host: '127.0.0.1',
  port: 5502,
  timeoutMs: 2000,
});

// ✅ v0.15.0 (New)
const transport = await AsyncTcpTransport.connect({
  host: '127.0.0.1',
  port: 5502,
  requestTimeoutMs: 2000,
});
```

---

## 2. Server Binding Methods are now Asynchronous

To prevent socket bind errors (e.g., port in use, EADDRINUSE) and serial port access errors from being silently swallowed, all server instantiation factory methods are now asynchronous. They must be awaited and will throw a standard `Error` if binding fails.

### Action Required
Add `await` to all server `bind`, `bindRtu`, and `bindAscii` calls, and handle potential bind exceptions.

```js
// ❌ v0.14.0 (Old)
const server = AsyncTcpModbusServer.bind(
  { host: '0.0.0.0', port: 5502 },
  handlers
);

// ✅ v0.15.0 (New)
try {
  const server = await AsyncTcpModbusServer.bind(
    { host: '0.0.0.0', port: 5502 },
    handlers
  );
  console.log('Server successfully bound to port 5502');
} catch (err) {
  console.error('Failed to bind server:', err.message);
}
```

This applies to:
- `AsyncTcpModbusServer.bind(...)`
- `AsyncSerialModbusServer.bindRtu(...)`
- `AsyncSerialModbusServer.bindAscii(...)`

---

## 3. Mandatory `unitId` for all Clients

The `createClient` method on **all** transports (`AsyncTcpTransport`, `AsyncRtuTransport`, and `AsyncAsciiTransport`) now strictly enforces passing a `unitId`. Omitting `unitId` or passing an empty object will result in a compile-time TypeScript error and a runtime exception.

### Action Required
Ensure that `unitId` is supplied when creating TCP or Serial clients.

```js
// ❌ v0.14.0 (Old)
const client = transport.createClient({}); // Allowed in TS typings, defaulted to 1 for TCP

// ✅ v0.15.0 (New)
const client = transport.createClient({ unitId: 1 }); // Strongly typed & required
```

---

## 4. `FifoQueueResponse.count` Removed from Types

The redundant `count` property in the return type of `readFifoQueue` (FC18) has been deprecated and removed from the public TypeScript interface. The length of the `values` array is always equivalent to the queue count.

### Action Required
Replace references to the `.count` property with `.values.length`.

```js
// ❌ v0.14.0 (Old)
const fifo = await client.readFifoQueue({ address: 0 });
console.log(`Queue size: ${fifo.count}`);

// ✅ v0.15.0 (New)
const fifo = await client.readFifoQueue({ address: 0 });
console.log(`Queue size: ${fifo.values.length}`);
```

---

## 5. Standardized `signal` Types

The `signal` parameters on all client read/write calls are now correctly typed as `AbortSignal` in the TypeScript declarations, instead of a generic `object`.

---

## New Features in v0.15.0

### 1. Robust Retry Configuration
You can now configure automatic retries and backoff strategies at connection time for all transports (`TcpTransportOptions`, `RtuTransportOptions`, and `AsciiTransportOptions`):

```js
const transport = await AsyncTcpTransport.connect({
  host: '127.0.0.1',
  port: 5502,
  requestTimeoutMs: 2000,
  retryAttempts: 3,                 // Retry up to 3 times before failing
  retryBackoffStrategy: 'fixed',    // 'immediate' | 'fixed' | 'exponential'
});
```

### 2. Stable Error Codes
You can now programmatically inspect errors using exported constants and a helper utility:

```js
const { getModbusErrorCode, ModbusErrorCode } = require('modbus-rs');

try {
  await client.readHoldingRegisters({ address: 0, quantity: 10 });
} catch (err) {
  const code = getModbusErrorCode(err);
  if (code === ModbusErrorCode.TIMEOUT) {
    console.error('Request timed out');
  } else if (code === ModbusErrorCode.EXCEPTION) {
    console.error('Modbus protocol exception occurred');
  }
}
```

### 3. Expanded Server Functionality
Four new request handlers have been added to the `ServerHandlers` callback interface:
- **FC07 (Read Exception Status):** `onReadExceptionStatus?: () => number | ModbusException | Promise<number | ModbusException>`
- **FC23 (Read/Write Multiple Registers):** `onReadWriteMultipleRegisters?: (req: ReadWriteMultipleRegistersRequest) => number[] | ModbusException | Promise<number[] | ModbusException>`
- **FC20 (Read File Record):** `onReadFileRecord?: (req: ReadFileRecordRequest) => number[][] | ModbusException | Promise<number[][] | ModbusException>`
- **FC21 (Write File Record):** `onWriteFileRecord?: (req: WriteFileRecordRequest) => void | ModbusException | Promise<void | ModbusException>`
