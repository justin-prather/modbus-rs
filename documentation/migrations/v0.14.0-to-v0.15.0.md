# Migration Guide: modbus-rs Node.js FFI (v0.14.0 â†’ v0.15.0)

This guide documents the breaking changes and new APIs introduced in **v0.15.0** of the `modbus-rs` Node.js bindings, and how to update your codebase.

---

## 1. TCP Connect Option Rename: `timeoutMs` â†’ `requestTimeoutMs`

To unify client transport options with the RTU and ASCII serial transports, the request timeout option has been renamed.

### Action Required
Update any calls to `AsyncTcpTransport.connect(...)` replacing the `timeoutMs` option with `requestTimeoutMs`.

```js
// âťŚ v0.14.0 (Old)
const transport = await AsyncTcpTransport.connect({
  host: '127.0.0.1',
  port: 5502,
  timeoutMs: 2000,
});

// âś… v0.15.0 (New)
const transport = await AsyncTcpTransport.connect({
  host: '127.0.0.1',
  port: 5502,
  requestTimeoutMs: 2000,
});
```

---

## 2. Server Binding Methods are now Asynchronous

To prevent socket bind errors (e.g., port in use, EADDRINUSE) and serial port access errors from being silently swallowed, all server instantiation factory methods are now asynchronous. They must be awaited and will throw a standard `Error` if binding fails.

### Action Required
Add `await` to all server `bind`, `bindRtu`, and `bindAscii` calls, and handle potential bind exceptions.

```js
// âťŚ v0.14.0 (Old)
const server = AsyncTcpModbusServer.bind(
  { host: '0.0.0.0', port: 5502 },
  handlers
);

// âś… v0.15.0 (New)
try {
  const server = await AsyncTcpModbusServer.bind(
    { host: '0.0.0.0', port: 5502 },
    handlers
  );
  console.log('Server successfully bound to port 5502');
} catch (err) {
  console.error('Failed to bind server:', err.message);
}
```

This applies to:
- `AsyncTcpModbusServer.bind(...)`
- `AsyncSerialModbusServer.bindRtu(...)`
- `AsyncSerialModbusServer.bindAscii(...)`

---

## 3. Mandatory `unitId` for all Clients

The `createClient` method on **all** transports (`AsyncTcpTransport`, `AsyncRtuTransport`, and `AsyncAsciiTransport`) now strictly enforces passing a `unitId`. Omitting `unitId` or passing an empty object will result in a compile-time TypeScript error and a runtime exception.

### Action Required
Ensure that `unitId` is supplied when creating TCP or Serial clients.

```js
// âťŚ v0.14.0 (Old)
const client = transport.createClient({}); // Allowed in TS typings, defaulted to 1 for TCP

// âś… v0.15.0 (New)
const client = transport.createClient({ unitId: 1 }); // Strongly typed & required
```

---

## 4. `FifoQueueResponse.count` Removed from Types

The redundant `count` property in the return type of `readFifoQueue` (FC18) has been deprecated and removed from the public TypeScript interface. The length of the `values` array is always equivalent to the queue count.

### Action Required
Replace references to the `.count` property with `.values.length`.

```js
// âťŚ v0.14.0 (Old)
const fifo = await client.readFifoQueue({ address: 0 });
console.log(`Queue size: ${fifo.count}`);

// âś… v0.15.0 (New)
const fifo = await client.readFifoQueue({ address: 0 });
console.log(`Queue size: ${fifo.values.length}`);
```

---

## 5. Standardized `signal` Types

The `signal` parameters on all client read/write calls are now correctly typed as `AbortSignal` in the TypeScript declarations, instead of a generic `object`.

## New Features in v0.15.0

### 1. Robust Retry Configuration
You can now configure automatic retries and backoff strategies at connection time for all transports (`TcpTransportOptions`, `RtuTransportOptions`, and `AsciiTransportOptions`):

```js
const transport = await AsyncTcpTransport.connect({
  host: '127.0.0.1',
  port: 5502,
  requestTimeoutMs: 2000,
  retryAttempts: 3,                 // Retry up to 3 times before failing
  retryBackoffStrategy: 'fixed',    // 'immediate' | 'fixed' | 'exponential'
});
```

### 2. Stable Error Codes
You can now programmatically inspect errors using exported constants and a helper utility:

```js
const { getModbusErrorCode, ModbusErrorCode } = require('modbus-rs');

try {
  await client.readHoldingRegisters({ address: 0, quantity: 10 });
} catch (err) {
  const code = getModbusErrorCode(err);
  if (code === ModbusErrorCode.TIMEOUT) {
    console.error('Request timed out');
  } else if (code === ModbusErrorCode.EXCEPTION) {
    console.error('Modbus protocol exception occurred');
  }
}
```

### 3. Expanded Server Functionality
Four new request handlers have been added to the `ServerHandlers` callback interface:
- **FC07 (Read Exception Status):** `onReadExceptionStatus?: () => number | ModbusException | Promise<number | ModbusException>`
- **FC23 (Read/Write Multiple Registers):** `onReadWriteMultipleRegisters?: (req: ReadWriteMultipleRegistersRequest) => number[] | ModbusException | Promise<number[] | ModbusException>`
- **FC20 (Read File Record):** `onReadFileRecord?: (req: ReadFileRecordRequest) => number[][] | ModbusException | Promise<number[][] | ModbusException>`
- **FC21 (Write File Record):** `onWriteFileRecord?: (req: WriteFileRecordRequest) => void | ModbusException | Promise<void | ModbusException>`

## 6. WASM Client Bindings Refactoring (Event-Driven Async Architecture & API Alignment)

In v0.15.0, the WASM client bindings have been completely re-architected. The old busy-polling loop (which ticked every 20ms using a timer) has been replaced by a modern, event-driven, async channel-based implementation. 

Alongside this structural rewrite, the public JavaScript/TypeScript API has been fully aligned with the Node.js bindings. This introduces major breaking changes to method names, parameter styles, transport instantiation, and return types.

### 6.1. Async Connection Method (`WasmTcpTransport.connect`)
The constructor `new WasmTcpTransport(url, options)` has been replaced with a static async factory method `WasmTcpTransport.connect(url, options)` that returns a Promise. This prevents race conditions and makes WebSocket connection error handling robust.

```js
// âťŚ v0.14.0 (Old)
const transport = new WasmTcpTransport("ws://127.0.0.1:8080", {
  responseTimeoutMs: 2000,
});
const client = transport.createClient();

// âś… v0.15.0 (New)
const transport = await WasmTcpTransport.connect("ws://127.0.0.1:8080", {
  responseTimeoutMs: 2000,
});
const client = transport.createClient({ unitId: 1 });
```

### 6.2. Mandatory `unitId` when Creating Clients
To align with Node.js behavior, `createClient` now strictly requires an options object containing `unitId`.

```js
// âťŚ v0.14.0 (Old)
const client = transport.createClient(); // unitId defaulted silently to 1

// âś… v0.15.0 (New)
const client = transport.createClient({ unitId: 1 }); // Required options object
```

### 6.3. Snake Case to Camel Case API & Options Objects
All client operations have been renamed from `snake_case` to `camelCase` and now take a single options object rather than multiple positional arguments.

```js
// âťŚ v0.14.0 (Old)
const coils = await client.read_coils(0, 10);
await client.write_multiple_registers(100, [1, 2, 3]);

// âś… v0.15.0 (New)
const coils = await client.readCoils({ address: 0, quantity: 10 });
await client.writeMultipleRegisters({ address: 100, values: [1, 2, 3] });
```

### 6.4. Normalized Return Types
Return types have been updated to be consistent with modern JavaScript standards and the Node.js API:

1. **Coils and Discrete Inputs:** Resolve to standard JavaScript `boolean[]` instead of bit-packed `Uint8Array`.
2. **Registers:** `readHoldingRegisters`, `readInputRegisters`, `readWriteMultipleRegisters`, and `readFifoQueue` **retain their memory-efficient `Uint16Array` return type** (they are NOT converted to regular JS numbers).
3. **Write/Ack Operations:** Write operations now return `Promise<void>` instead of acknowledgment objects like `{ address, value }` or `{ address, quantity }`.

| Method | Old Return Shape | New Return Shape (v0.15.0) |
| :--- | :--- | :--- |
| `readCoils` / `readDiscreteInputs` | `Uint8Array` (bit-packed) | `boolean[]` |
| `readHoldingRegisters` / `readInputRegisters` | `Uint16Array` | `Uint16Array` |
| `readWriteMultipleRegisters` | `Uint16Array` | `Uint16Array` |
| `readFifoQueue` | `Uint16Array` | `Uint16Array` |
| `writeSingleCoil` / `writeMultipleCoils` | `{ address, value }` or `{ address, quantity }` | `void` |
| `writeSingleRegister` / `writeMultipleRegisters` | `{ address, value }` or `{ address, quantity }` | `void` |

### 6.5. Removed Single-Item Read Methods
Granular helper methods that read a single coil or register have been removed. Use the standard vector read methods with `quantity: 1` instead.

**Removed methods:**
- `readSingleCoil` (use `readCoils({ address, quantity: 1 })`)
- `readSingleDiscreteInput` (use `readDiscreteInputs({ address, quantity: 1 })`)
- `readSingleHoldingRegister` (use `readHoldingRegisters({ address, quantity: 1 })`)
- `readSingleInputRegister` (use `readInputRegisters({ address, quantity: 1 })`)

### 6.6. Support for `maskWriteRegister`
A new method `maskWriteRegister` has been added to the WASM client API. It accepts options for `address`, `andMask`, and `orMask`:

```js
await client.maskWriteRegister({
  address: 10,
  andMask: 0x00FF,
  orMask: 0xFF00,
});
```

## 7. WASM Server Bindings Refactoring (Event-Driven Async Architecture & API Alignment)

In v0.15.0, the WASM server bindings (`WasmTcpServer` and `WasmSerialServer`) have been completely re-architected to align with the Node.js async server pattern. The old synchronous startup/shutdown methods, manual frame polling, and single-function request dispatch have been replaced with a fully self-contained, event-driven async server event loop and a structured, callback-based request dispatching system.

### 7.1. Async Server Binding Factories

In v0.14.0, servers were instantiated synchronously via `new WasmTcpServer(...)` or `new WasmSerialServer(...)` and started synchronously using a `.start()` method.

In v0.15.0, servers are instantiated and bound asynchronously via static factory methods returning a `Promise` that resolves to the server instance once successfully bound and connected.

```js
// âťŚ v0.14.0 (Old)
const server = new WasmTcpServer(
  new WasmTcpGatewayConfig("ws://127.0.0.1:8080"),
  (req) => { /* ... */ }
);
server.start();

// âś… v0.15.0 (New)
const server = await WasmTcpServer.bind(
  {
    wsUrl: "ws://127.0.0.1:8080",
    unitId: 1
  },
  handlers
);
server.serve().catch(err => console.error("Server event loop error:", err));
```

For Serial servers, they are bound using RTU or ASCII static methods:

```js
// âťŚ v0.14.0 (Old)
const config = WasmSerialServerConfig.rtu();
const server = new WasmSerialServer(config, (req) => { /* ... */ });
server.attach_serial_port(serialPort);
server.start();

// âś… v0.15.0 (New)
const server = await WasmSerialServer.bindRtu(
  {
    serialPort: serialPort,
    unitId: 1,
    baudRate: 9600,
  },
  handlers
);
server.serve().catch(err => console.error("Server event loop error:", err));
```

### 7.2. Elimination of Manual Frame Polling & Lower-level API Removal

The new event-driven architecture processes Modbus frames entirely in a background async driver task. The following low-level methods and configuration types have been removed from the public JS/TS interface:

- `start()` / `stop()` (replaced by async `bind` and `shutdown()`)
- `send_frame(data)`
- `recv_frame()`
- `dispatch_request(req)`
- `status_snapshot()` / `WasmServerStatusSnapshot`
- `last_error_message()` / `clear_last_error()`
- `WasmTcpGatewayConfig` / `WasmSerialServerConfig` / `WasmServerBindingPlan` (replaced by options objects)

### 7.3. Structured Callback Handler (`ServerHandlers`)

In v0.14.0, the handler argument was a single opaque `Function` receiving a generic request object, requiring you to manually check function codes and construct response structures.

In v0.15.0, the server accepts a `ServerHandlers` object containing callback methods mapped to specific Modbus Function Codes (FCs), aligning perfectly with the Node.js API. These callbacks can return direct values or `Promise`s.

```js
// âťŚ v0.14.0 (Old)
const server = new WasmTcpServer(config, (req) => {
  if (req.fc === 3) {
    return [10, 20, 30];
  }
  // ...
});

// âś… v0.15.0 (New)
const handlers = {
  onReadHoldingRegisters: (req) => {
    console.log(`Reading registers at address ${req.address} for unit ${req.unitId}`);
    return [10, 20, 30]; // Can also be Promise<number[]>
  },
  onWriteSingleRegister: (req) => {
    console.log(`Writing ${req.value} to address ${req.address}`);
  }
};
const server = await WasmTcpServer.bind({ wsUrl, unitId: 1 }, handlers);
```

### 7.4. Server Event Loop Execution (`serve`)

In v0.15.0, creating and binding a server does not automatically run the server's background request-processing event loop. To begin handling incoming client connections and requests, you must invoke `.serve()`.

This method returns a `Promise<void>` that remains pending for the duration of the server lifecycle, resolving on graceful `.shutdown()` and rejecting if the connection/session crashes.

```js
// Start the event loop (runs asynchronously in background)
server.serve().catch(err => {
  console.error("Server crashed:", err);
});
```

### 7.5. Async Server Teardown (`shutdown`)

The synchronous `.stop()` method is replaced by an async `.shutdown()` method returning a `Promise` that resolves when the background task successfully terminates and stops processing frames.

```js
// âťŚ v0.14.0 (Old)
server.stop();

// âś… v0.15.0 (New)
await server.shutdown();
```
