# Migration Guide: v0.14.0 (from v0.13.0)

This migration guide details the steps required to migrate your Node.js and Python codebases to the new **Transport + Client Factory** API pattern introduced in `modbus-rs` v0.14.0.

---
## Nodejs v0.14.0 (from v0.13.0)

#### Breaking Changes Summary

| Area | Old (v0.13.0) | New (v0.14.0) | Action |
|---|---|---|---|
| **TCP Client Instantiation** | `AsyncTcpModbusClient.connect(opts)` | `AsyncTcpTransport.connect(opts)` | Change static connection method to instantiate the transport first. |
| **Serial Client Instantiation** | `AsyncSerialModbusClient.connectRtu(opts)` / `connectAscii()` | `AsyncRtuTransport.open(opts)` / `AsyncAsciiTransport.open()` | Change static connection methods to open the transport first. |
| **Client Construction** | Connection options contained `unitId` | Client created via `transport.createClient({ unitId })` | Remove `unitId` from transport connection options; call `.createClient()` to get a logical handle. |
| **Lifecycle Methods** | `close()`, `reconnect()`, `setRequestTimeout()`, `pendingRequests` on client | Reside exclusively on the **Transport** instance | Invoke connection control and lifecycle methods on the transport. |
| **Options Types** | `TcpClientOptions`, `SerialClientOptions` | `TcpTransportOptions`, `RtuTransportOptions`, `AsciiTransportOptions` | Rename option configuration structures and remove `unitId` from transport configurations. |

---

#### Node.js Migration

This redesign splits physical connection management (Transports) from logical target devices (Clients), enabling multi-drop configurations (sharing a single serial port or TCP gateway across multiple unit IDs).

##### 1. Migrating Modbus TCP Client

###### Before (v0.13.0 and earlier)
A single client class managed both the TCP connection socket and Modbus requests for a single unit ID:

```javascript
const { AsyncTcpModbusClient } = require('modbus-rs');

// Connect and build the client directly
const client = await AsyncTcpModbusClient.connect({
  host: '127.0.0.1',
  port: 502,
  unitId: 1,         // Combined connection & device identity
  timeoutMs: 2000,
});

// Run commands
const regs = await client.readHoldingRegisters({ address: 0, quantity: 10 });

// Close the connection through the client
await client.close();
```

###### After (v0.14.0+)
Separate the physical socket (`AsyncTcpTransport`) from the logical device client (`AsyncTcpModbusClient`):

```javascript
const { AsyncTcpTransport } = require('modbus-rs');

// 1. Establish the physical socket transport
const transport = await AsyncTcpTransport.connect({
  host: '127.0.0.1',
  port: 502,
  timeoutMs: 2000,   // unitId is no longer here
});

// 2. Create the logical client handle (synchronous, no I/O)
// unitId is optional for TCP and defaults to 1.
const client = transport.createClient({ unitId: 1 });

// 3. Run commands (identical API)
const regs = await client.readHoldingRegisters({ address: 0, quantity: 10 });

// 4. Close/reconnect/configure timeouts via the transport
await transport.close();
```

##### 2. Migrating Modbus Serial Client (RTU/ASCII)

###### Before (v0.13.0 and earlier)
A single serial client class opened the serial port and locked the unit ID:

```javascript
const { AsyncSerialModbusClient } = require('modbus-rs');

// Opened the serial port and bound a unitId
const client = await AsyncSerialModbusClient.connectRtu({
  portPath: '/dev/ttyUSB0',
  baudRate: 115200,
  parity: 'none',
  unitId: 1,
  requestTimeoutMs: 1000,
});

// Run commands
const regs = await client.readHoldingRegisters({ address: 0, quantity: 10 });

// Closed the serial port
await client.close();
```

###### After (v0.14.0+)
Open the serial port once using the appropriate transport (`AsyncRtuTransport` or `AsyncAsciiTransport`) and spawn a client bound to a specific `unitId` (required for serial):

```javascript
const { AsyncRtuTransport } = require('modbus-rs');

// 1. Open the physical serial port
const transport = await AsyncRtuTransport.open({
  portPath: '/dev/ttyUSB0',
  baudRate: 115200,
  parity: 'none',
  requestTimeoutMs: 1000,  // unitId is no longer here
});

// 2. Derive the client handle (synchronous, no I/O)
const client = transport.createClient({ unitId: 1 });

// 3. Run commands
const regs = await client.readHoldingRegisters({ address: 0, quantity: 10 });

// 4. Close the serial port via the transport
await transport.close();
```

##### 3. Multi-Drop Topology Example
This redesign allows communicating with multiple slave devices on a single physical serial port or TCP-to-RTU gateway without connection conflicts.

```javascript
const { AsyncRtuTransport } = require('modbus-rs');

const transport = await AsyncRtuTransport.open({
  portPath: '/dev/ttyUSB0',
  baudRate: 9600,
});

// Create multiple client handles pointing to different unit IDs on the same bus
const deviceA = transport.createClient({ unitId: 1 });
const deviceB = transport.createClient({ unitId: 2 });
const deviceC = transport.createClient({ unitId: 3 });

// Read from deviceA, then deviceB
// (The shared transport automatically sequences requests to prevent collisions)
const tempA = await deviceA.readHoldingRegisters({ address: 100, quantity: 1 });
const tempB = await deviceB.readHoldingRegisters({ address: 100, quantity: 1 });

console.log(`Device A: ${tempA[0]}, Device B: ${tempB[0]}`);

await transport.close(); // Closes the serial port for all derived clients
```

|---|---|---|

## Python v0.14.0 (from v0.13.0)

#### Breaking Changes Summary

| Area | Old (v0.13.0) | New (v0.14.0) | Action |
|---|---|---|---|
| **TCP Client Instantiation** | `TcpClient(host, port, unit_id, ...)` / `AsyncTcpClient(...)` | `TcpTransport.connect(host, port, ...)` / `AsyncTcpTransport.connect(...)` | Change constructor call to connect/establish the transport first. |
| **Serial Client Instantiation** | `SerialClient(port, baud_rate, unit_id, ...)` / `AsyncSerialClient(...)` | `RtuTransport.open(port, baud_rate, ...)` / `AsyncRtuTransport.open(...)` | Change constructor call to open the transport first. |
| **Client Construction** | Client constructor contained `unit_id` | Client created via `transport.create_client(unit_id)` | Remove `unit_id` from transport connection parameters; call `.create_client(unit_id)` on transport to get client. |
| **Lifecycle Methods** | `close()`, `timeout` / `request_timeout` on client | Reside on the **Transport** instance, or handled via context managers (`with` / `async with`) | Invoke connection control and lifecycle methods on the transport. |

#### Python Migration

This redesign splits physical connection management (Transports) from logical target devices (Clients), enabling multi-drop configurations (sharing a single serial port or TCP gateway/connection across multiple unit IDs).

##### 1. Migrating Modbus TCP Client

###### Before (v0.13.0 and earlier)
A single client class managed both the TCP connection socket and Modbus requests for a single unit ID:

```python
import modbus_rs

# Connect and build the client directly
client = modbus_rs.TcpClient(
    host="127.0.0.1",
    port=502,
    unit_id=1,         # Combined connection & device identity
    timeout_ms=2000,
)

# Run commands
regs = client.read_holding_registers(0, 10)

# Close the connection through the client
client.close()
```

###### After (v0.14.0+)
Separate the physical socket (`TcpTransport` / `AsyncTcpTransport`) from the logical device client (`TcpModbusClient` / `AsyncTcpModbusClient`). Transports are designed to be used as context managers:

```python
import modbus_rs

# 1. Establish the physical socket transport (unit_id is no longer here)
with modbus_rs.TcpTransport.connect("127.0.0.1", port=502, timeout_ms=2000) as transport:
    # 2. Create the logical client handle (synchronous, no I/O)
    client = transport.create_client(unit_id=1)

    # 3. Run commands (identical API)
    regs = client.read_holding_registers(0, 10)

# 4. Connection is closed automatically when exiting the 'with' block
```

##### 2. Migrating Modbus Serial Client (RTU/ASCII)

###### Before (v0.13.0 and earlier)
A single serial client class opened the serial port and locked the unit ID:

```python
import modbus_rs

# Opened the serial port and bound a unit_id
client = modbus_rs.SerialClient(
    port="/dev/ttyUSB0",
    baud_rate=115200,
    unit_id=1,
    timeout_ms=1000,
)

# Run commands
regs = client.read_holding_registers(0, 10)

# Closed the serial port
client.close()
```

###### After (v0.14.0+)
Open the serial port once using the appropriate transport (`RtuTransport` or `AsciiTransport`) and spawn a client bound to a specific `unit_id`:

```python
import modbus_rs

# 1. Open the physical serial port
with modbus_rs.RtuTransport.open("/dev/ttyUSB0", baud_rate=115200, timeout_ms=1000) as transport:
    # 2. Derive the client handle (synchronous, no I/O)
    client = transport.create_client(unit_id=1)

    # 3. Run commands
    regs = client.read_holding_registers(0, 10)

# 4. Serial port is closed automatically when exiting the 'with' block
```

##### 3. Migrating Async TCP/Serial Client

For `asyncio`-based applications, use the `Async` counterparts. Note that `AsyncTcpTransport.connect` / `AsyncRtuTransport.open` are async functions and their return values are async context managers:

```python
import asyncio
import modbus_rs

async def main():
    # Establish async transport connection (awaits connection, returns context manager)
    async with await modbus_rs.AsyncTcpTransport.connect("127.0.0.1", port=502) as transport:
        # Create client handle
        client = transport.create_client(unit_id=1)
        
        # Run async commands
        regs = await client.read_holding_registers(0, 10)
        print(regs)

asyncio.run(main())
```

##### 4. Multi-Drop Topology Example
This redesign allows communicating with multiple slave devices on a single physical serial port or TCP gateway without connection conflicts.

```python
import modbus_rs

with modbus_rs.RtuTransport.open("/dev/ttyUSB0", baud_rate=9600) as transport:
    # Create multiple client handles pointing to different unit IDs on the same bus
    device_a = transport.create_client(unit_id=1)
    device_b = transport.create_client(unit_id=2)
    device_c = transport.create_client(unit_id=3)

    # Read from device_a, then device_b
    temp_a = device_a.read_holding_registers(100, 1)
    temp_b = device_b.read_holding_registers(100, 1)

    print(f"Device A: {temp_a[0]}, Device B: {temp_b[0]}")
```

## WebAssembly (WASM) v0.14.0 (from v0.13.0)

#### Packaging and Import Migration

In `modbus-rs-wasm` v0.14.0, the package structure and exports have been modernized to follow Node.js standard package entry conditions.

##### 1. Removal of Vite Alias Workaround
In previous versions, a Vite project might have required a resolve alias in `vite.config.ts` to load the WASM distribution package correctly:

```typescript
// vite.config.ts
export default defineConfig({
  resolve: {
    alias: {
      // ❌ No longer needed in v0.14.0!
      "modbus-rs-wasm": resolve(__dirname, "../../node_modules/modbus-rs-wasm/dist/web/modbus-rs.js"),
    },
  },
});
```

This alias is **no longer required** and should be removed.

##### 2. Explicit Subpath Exports and Condition Resolution
The `modbus-rs-wasm` packaging now properly separates targeting for browser and node runtimes via custom conditions and provides wildcard subpath support for custom importing needs.

- **Wildcard Subpath Exports**: You can now import directly from dist folders without bundler blocking:
  ```javascript
  import init from "modbus-rs-wasm/dist/web/modbus-rs.js";
  ```
- **Proper Bundler Targets**: The missing `dist/bundler/modbus-rs_bg.js` helper glue file has been added to the package manifest, fixing all standard ESM and bundler builds without requiring manual configuration workarounds.
