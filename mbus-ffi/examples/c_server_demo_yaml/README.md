# C Server Demo: YAML-Driven Code Generation

This example shows the full workflow for building a Modbus TCP server in C using
the `mbus-ffi` static library.  A YAML file declares the device register map;
`xtask` generates the Rust handler table and a matching C header; `main.c`
implements application logic and transport, then wires everything together with
`mbus_tcp_server_new`.

## Architecture

```
┌──────────────────────────────────────────────────────────┐
│  main.c  (your C application)                            │
│                                                          │
│  • Implements write-notification hooks (app_on_write_*)  │
│  • Provides real TCP transport callbacks                 │
│  • Calls mbus_server_set_* to update read-only registers │
│  • Runs a server poll thread (pthread)                   │
├──────────────────────────────────────────────────────────┤
│  mbus_server_app.h  (generated C header)                 │
│  • mbus_server_model_init()                              │
│  • mbus_server_default_handlers(userdata)                │
│  • mbus_server_get_* / mbus_server_set_* per register    │
│  • app_on_write_* hook declarations                      │
├──────────────────────────────────────────────────────────┤
│  generated_server.rs  (generated by build.rs → $OUT_DIR) │
│  • APP_MODEL static (register/coil storage)              │
│  • MbusServerHandlers callback table                     │
│  • Named getter/setter FFI exports                       │
│  • Never tracked in git                                  │
├──────────────────────────────────────────────────────────┤
│  mbus-ffi static library  (libmbus_ffi.a)                │
│  • mbus_tcp_server_new / connect / poll / disconnect     │
│  • Modbus protocol engine                                │
└──────────────────────────────────────────────────────────┘
```

## Files

| File | Purpose |
|---|---|
| `mbus_server_app.example.yaml` | Device register map declaration |
| `main.c` | Transport, hooks, threading, Industrial application server lifecycle |
| `mbus_server_app.h` | Generated C header (do not edit by hand) |
| `CMakeLists.txt` | Build config — always links `libmbus_ffi.a` statically |
| `CMakePresets.json` | Presets for macOS (forces Apple Clang) user may reconfigure for your system requirements |

## Step 1: Define the Device in YAML

`mbus_server_app.example.yaml` declares every register and coil along with an
optional write-notification hook that Rust calls before storing the new value:

```yaml
schema_version: 1

device:
  name: PumpController
  slave_address: 1
  response_timeout_ms: 1000

memory_map:
  coils:
    - name: pump_run
      address: 0
      access: rw
      on_write: app_on_write_pump_run     # called before value is stored

    - name: alarm_ack
      address: 1
      access: rw
      on_write: app_on_write_alarm_ack

  discrete_inputs:
    - name: fault_active
      address: 0
      access: ro                          # read-only; no on_write

  holding_registers:
    - name: speed_setpoint
      address: 0
      access: rw
      on_write: app_on_write_speed_setpoint

    - name: pressure_limit
      address: 1
      access: rw
      on_write: app_on_write_pressure_limit

  input_registers:
    - name: pressure_actual
      address: 0
      access: ro

    - name: temperature_actual
      address: 1
      access: ro
```

## Step 2: Generate Code

```bash
cargo run -p xtask -- build-c-demo c_server_demo_yaml --static
```

This single command (xtask reads `demo.yaml` and sets `MBUS_SERVER_APP_CONFIG` automatically):
1. Runs `gen-server-app` → writes `mbus_server_app.h` (C header only)
2. Compiles `libmbus_ffi.a` — `build.rs` reads `MBUS_SERVER_APP_CONFIG` and generates
   `generated_server.rs` into `$OUT_DIR` (never written to the source tree)
3. Configures CMake with `build-static/` and links statically
4. Runs CTest (`--self-test` mode, no network required)

To regenerate only the C header (without building):

```bash
cargo run -p xtask -- gen-server-app \
  --config mbus-ffi/examples/c_server_demo_yaml/mbus_server_app.example.yaml \
  --emit-c-header mbus-ffi/include/mbus_server_app.h
```

Verify the checked-in C header is in sync with the YAML:

```bash
cargo run -p xtask -- check-server-gen
```

> **Note:** The `generated_server.rs` Rust dispatcher is generated at compile time by
> `mbus-ffi/build.rs` via the `MBUS_SERVER_APP_CONFIG` environment variable.  It is
> written into Cargo's `$OUT_DIR` and is never committed to git.  You do not need to
> run `gen-server-app` to update it — a normal `cargo build` is sufficient.

## Step 3: What Gets Generated

### `mbus_server_app.h` (C header)

```c
// Initialise the Rust-owned register/coil model (call once at startup).
void mbus_server_model_init(void);

// Return a fully-populated MbusServerHandlers table.
// Pass a pointer to your AppState; it is forwarded to every on_write hook.
struct MbusServerHandlers mbus_server_default_handlers(void *userdata);

// Named accessors — thread-safe (acquire s_app_mutex internally).
void     mbus_server_set_speed_setpoint(uint16_t v);
uint16_t mbus_server_get_speed_setpoint(void);
void     mbus_server_set_pressure_actual(uint16_t v);
// ... one pair per register/coil ...

// Write-notification hook declarations (you implement these in main.c).
MbusHookStatus app_on_write_pump_run(void *ctx, uint16_t address, uint8_t value);
MbusHookStatus app_on_write_speed_setpoint(void *ctx, uint16_t address, uint16_t value);
// ...
```

### `generated_server.rs` (Rust, generated by `build.rs` at compile time)

This file is **not** present in the source tree.  `mbus-ffi/build.rs` reads
`MBUS_SERVER_APP_CONFIG` and writes the file to Cargo's `$OUT_DIR`; `mod.rs`
includes it with the `include!` macro.  It contains:

- `APP_MODEL` — static storage for all registers and coils, protected by `mbus_app_lock`
- One `extern "C"` handler per function code (FC01 read coils, FC03 read holding regs, FC05 write coil, FC06 write register, …)
- Calls the appropriate `on_write` hook before storing writes; returns a Modbus exception if the hook rejects the value

To trigger regeneration, simply rebuild with `MBUS_SERVER_APP_CONFIG` set — no
separate codegen step needed.

## Step 4: Implement Hooks in C

In `main.c` you provide the write-notification hooks declared in the YAML.
These are called by Rust **before** the value is stored in `APP_MODEL`.

```c
// IMPORTANT: do NOT call mbus_server_get_* / mbus_server_set_* from here.
// Rust already holds s_app_mutex at the call site.

MbusHookStatus app_on_write_pump_run(void *ctx, uint16_t address, uint8_t value) {
    (void)address;
    struct AppState *app = (struct AppState *)ctx;
    if (!app) return MBUS_HOOK_DEVICE_FAILURE;
    app->pump_running = (value != 0u);
    printf("[write] pump_run = %s\n", app->pump_running ? "ON" : "OFF");
    return MBUS_HOOK_OK;
}

MbusHookStatus app_on_write_speed_setpoint(void *ctx, uint16_t address, uint16_t value) {
    (void)address;
    struct AppState *app = (struct AppState *)ctx;
    if (!app) return MBUS_HOOK_DEVICE_FAILURE;
    if (value > 5000u) return MBUS_HOOK_ILLEGAL_DATA_VALUE; /* reject out-of-range */
    printf("[write] speed_setpt = %u RPM\n", value);
    return MBUS_HOOK_OK;
}
```

Return values:

| Value | Meaning |
|---|---|
| `MBUS_HOOK_OK` | Accept — Rust stores the value |
| `MBUS_HOOK_ILLEGAL_DATA_VALUE` | Reject — server replies with Modbus exception 03 |
| `MBUS_HOOK_ILLEGAL_DATA_ADDRESS` | Reject — server replies with Modbus exception 02 |
| `MBUS_HOOK_DEVICE_FAILURE` | Reject — server replies with Modbus exception 04 |

## Step 5: Wire the Server

```c
// 1. Initialise register storage.
mbus_server_model_init();

// 2. Seed initial values for read-only registers.
mbus_server_set_speed_setpoint(1500u);
mbus_server_set_pressure_actual(120u);

// 3. Build the transport callbacks (real TCP or custom).
struct MbusTransportCallbacks tr = {
    .userdata        = &tctx,
    .on_connect      = tcp_connect,
    .on_disconnect   = tcp_disconnect,
    .on_send         = tcp_send,
    .on_recv         = tcp_recv,
    .on_is_connected = tcp_is_connected,
};

// 4. Get the generated handler table.
struct MbusServerHandlers handlers = mbus_server_default_handlers(&app);

// 5. Create the server.
struct MbusServerConfig cfg = { .slave_address = 1u, .response_timeout_ms = 1000u };
MbusServerId id = mbus_tcp_server_new(&tr, &handlers, &cfg);

// 6. Connect transport, start poll thread, run application loop.
mbus_tcp_server_connect(id);
// ... pthread_create(server_thread_fn) ...
// ... application loop calls mbus_server_set_* to update live sensor values ...
```

### Lock shims

The application must provide implementations for the mutex hooks that Rust calls
internally.  Use real `pthread_mutex_t` in any multi-threaded application:

```c
static pthread_mutex_t s_server_mutex = PTHREAD_MUTEX_INITIALIZER;

// Called by Rust — do NOT call from application code.
void mbus_server_lock(MbusServerId id)   { (void)id; pthread_mutex_lock(&s_server_mutex); }
void mbus_server_unlock(MbusServerId id) { (void)id; pthread_mutex_unlock(&s_server_mutex); }
```

**Never call `mbus_server_lock` / `mbus_server_unlock` directly** — every
`mbus_tcp_server_*` function acquires the lock internally via RAII.  Calling it
from application code would double-lock (deadlock with a non-recursive mutex).

## Running the Demo

### Two modes

| Mode | Command | Use case |
|---|---|---|
| Real TCP server | `./c_server_demo_yaml` | Connect any Modbus TCP client to port 5020 |
| Self-test | `./c_server_demo_yaml --self-test` | In-process verification, no network needed |

CTest always uses `--self-test`.

### Build and test

```bash
# From workspace root — generates C header, builds libmbus_ffi.a, and runs CTest:
cargo run -p xtask -- build-c-demo c_server_demo_yaml --static
```

### Manual build (macOS)

```bash
cd mbus-ffi/examples/c_server_demo_yaml
cmake --preset macos-static   # configure into build-static/
cmake --build --preset macos-static
./build-static/c_server_demo_yaml --self-test
```

### Run as a real server

```bash
./build-static/c_server_demo_yaml
# Modbus TCP server listening on 0.0.0.0:5020  (unit id 1)
# Connect with any Modbus TCP client (e.g. ModbusLab, mbpoll, pymodbus)
# Press Ctrl-C to stop.
```

### Register map (unit id 1, port 5020)

| Type | Address | Name | FC |
|---|---|---|---|
| Coil | 0x0000 | pump_run | FC01 / FC05 |
| Coil | 0x0001 | alarm_ack | FC01 / FC05 |
| Discrete input | 0x0000 | fault_active | FC02 |
| Holding register | 0x0000 | speed_setpoint | FC03 / FC06 |
| Holding register | 0x0001 | pressure_limit | FC03 / FC06 |
| Input register | 0x0000 | pressure_actual | FC04 |
| Input register | 0x0001 | temperature_actual | FC04 |
